# Flatpack specification

The brutally specific version. If a tool calls itself a Flatpack and
contradicts anything in this document, it isn't one.

Nothing in here is aspirational. Everything is a constraint.

---

## 1. What is a Flatpack?

> **A Flatpack is a single-file, local-first HTML tool generated by an
> AI coding agent, designed to be saved to disk, reopened, sent, edited
> by a different agent later, with no backend, no build step, no
> default network calls, and an explicit, inspectable internal
> structure.**

Three load-bearing claims, in order of importance:

1. **Agent-editable.** The internal structure exists so a different
   agent, in a different session, can modify the file without
   re-architecting it. This is the rarest property and the one to lead
   with. Without it, Flatpacks are disposable HTML; with it, they
   compound.
2. **Local-first.** Saved to disk, runs offline, owns its data, leaves
   the chat window. This is the survival property — it makes a Flatpack
   useful six months from now.
3. **Single-file.** The artifact is the file. No build step, no
   bundler, no install. This is the distribution property — it lets
   anyone open or send the tool with no setup.

A Flatpack is not a generator. It is not a platform. It is not a
hosted product. It is the **format** that an AI agent produces.

### Templates are not a framework

There is no `extend`, `include`, or shared runtime. The four canonical
templates are copy-edit starting points, not parents. **The shape is
the API.** Reuse happens by copying a template and changing the
schema, the core logic, and the rendering — never by importing or
inheriting. This is what keeps the form factor cheap.

---

## 2. What does the user receive?

One of two things, never both.

### Mode A — Single-file Flatpack (default)

```
index.html
```

Embedded inside that one file:

- HTML structure
- CSS (including print styles)
- JavaScript (state, schema, validation, core logic, rendering, events, tests)
- Sample data
- Help text and privacy disclosure
- AI editing notes

The user double-clicks the file. The tool runs.

### Mode B — Packet Flatpack

```
my-tool/
  index.html            # the actual Flatpack
  README.md             # human-readable purpose, usage, limitations
  sample-data.csv       # optional
  examples/             # optional
```

Used when the tool legitimately needs sample files alongside it. The
**tool itself is still a single HTML file**. The folder is for
companion materials only — never code that the HTML file loads.

---

## 3. What does it run on?

- A modern browser opened directly via `file://` or a static host.
  Tested against the last two stable major versions of Chrome, Safari,
  Firefox, and Edge.
- No Node, no Python, no Docker, no `npm install`.
- No server. No build step. No bundler.
- Standards-track browser APIs only: HTML, CSS, ES2020+ JavaScript,
  the File API, `localStorage`, `IndexedDB`, Web Workers, `<canvas>`,
  SVG, the Print CSS spec, `Blob` + `URL.createObjectURL`.

If a Flatpack does not work offline, it is not a Flatpack.

---

## 4. What is forbidden?

These are non-negotiable. A reviewer should reject any tool that
violates them.

- **No external resources.** No `<script src="https://...">`. No
  `<link href="https://...">`. No remote fonts, no CDN icons, no
  remote images. If you need an icon, inline an SVG.
- **No network calls.** No `fetch()`, `XMLHttpRequest`, `WebSocket`,
  `sendBeacon`, `EventSource`, dynamic `<img>` to a remote host, none
  of it. By default. The exception is a tool whose explicit purpose is
  to call a user-provided endpoint, and even then it must declare this
  in `APP_META.network` and ask for the URL at runtime — never embed it.
- **No embedded secrets.** No API keys, no tokens, no passwords. If
  the tool needs one, the user pastes it into the page at runtime and
  it lives only in `sessionStorage` or memory.
- **No telemetry.** No analytics. No error reporting. No "phone home."
- **No dependencies.** No npm imports. No CDN scripts. No CSS frameworks
  loaded over the wire. A Flatpack carries everything it needs inside
  itself.
- **No build step.** The file the user saves is the file the user
  opens. No compilation, no minification, no transpile.
- **No false claims.** A Flatpack must not describe itself as "secure,"
  "encrypted," or "compliant" unless it actually implements those
  controls and can demonstrate them.

## 5. What persistence is allowed?

Three levels. Every Flatpack picks one and discloses it.

| Level | API | When to use |
|---|---|---|
| `none` | — | Quick calculators, throwaway converters, privacy-sensitive sessions. |
| `localStorage` | `localStorage.*` | Form inputs, settings, recent results, checklist progress. Small data. |
| `indexedDB` | `indexedDB.*` | Imported CSVs, document metadata, larger dashboards, local review queues. |

The chosen level is named in `APP_META.persistence` and is reflected
verbatim in the privacy banner that the user sees on first load.

Persistence rules:

- **Disclose, always.** No hidden writes.
- **Versioned keys.** Use `flatpack:<tool-slug>:v1` so a future shape
  change is a deliberate migration, not silent corruption.
- **Degrade gracefully.** `localStorage` is unavailable in private
  browsing on some browsers. The tool must still work without it,
  losing only state-persistence between sessions.
- **Provide reset.** Every Flatpack with persistence must have a
  visible "Reset" control that clears it.

## 6. What must every Flatpack include?

In the file, in this order, marked by HTML comments so a coding
agent can find each region by exact-match grep:

```html
<!--
  Flatpack: <Tool name>
  Version: 0.1.0
  Purpose: <One sentence>
  Data: <local-only | none>
  Network: <none | user-supplied-endpoint>
-->

<!-- FLATPACK:HELP -->           help text + AI editing notes (see §6.2)
<!-- FLATPACK:STYLES -->         CSS, including print styles
<!-- FLATPACK:HTML -->           static DOM
<!-- FLATPACK:TEMPLATES -->      optional: HTML fragments
<!-- FLATPACK:MANIFEST -->       machine-readable manifest (see §6.1)
<!-- FLATPACK:METADATA -->       APP_META constant
<!-- FLATPACK:SCHEMA -->         shape of inputs and records
<!-- FLATPACK:STATE -->          centralised state + load/save
<!-- FLATPACK:VALIDATION -->     single validate() function
<!-- FLATPACK:IMPORT_EXPORT -->  CSV/JSON/Markdown/clipboard helpers
<!-- FLATPACK:CORE_LOGIC -->     pure functions, no DOM, no state
<!-- FLATPACK:RENDERING -->      DOM updates from state, deterministic
<!-- FLATPACK:EVENTS -->         input handlers that mutate state + call render()
<!-- FLATPACK:TEST_CASES -->     inline test cases for core logic
<!-- FLATPACK:BOOT -->           DOMContentLoaded startup
```

### 6.1 The promotion manifest

Every Flatpack carries a machine-readable manifest as an inline JSON
block. It is what makes promotion possible without magic — a future
agent reads it to plan the Baseplate version.

```html
<!-- FLATPACK:MANIFEST -->
<script type="application/json" id="flatpack-manifest">
{
  "name": "Invoice Cleaner",
  "artifactType": "csv-cleaner",
  "version": "0.2.0",
  "scope": "personal",
  "persistence": "localStorage",
  "network": "none",
  "runtime": {
    "minimumBrowser": { "chrome": 100, "safari": 15.4, "firefox": 98, "edge": 100 }
  },
  "entities": [
    {
      "name": "Invoice",
      "fields": [
        { "name": "supplierName",  "type": "string", "required": true },
        { "name": "invoiceNumber", "type": "string", "required": true, "unique": true },
        { "name": "invoiceDate",   "type": "date",   "required": true },
        { "name": "amount",        "type": "number", "required": true, "gt": 0 },
        { "name": "currency",      "type": "enum",   "values": ["GBP","EUR","USD"], "default": "GBP" }
      ]
    }
  ],
  "validations": [
    "invoiceNumber is required",
    "amount must be greater than zero",
    "invoiceDate must be a valid date and not in the future",
    "duplicate invoice numbers should be flagged",
    "currency outside the allowed list is a warning, defaults to GBP"
  ],
  "validation_predicates": [
    { "field": "invoiceNumber", "constraint": "required" },
    { "field": "invoiceNumber", "constraint": "unique" },
    { "field": "amount",        "constraint": "gt", "value": 0 },
    { "field": "invoiceDate",   "constraint": "format", "value": "date" },
    { "field": "invoiceDate",   "constraint": "not_in_future" },
    { "field": "currency",      "constraint": "one_of", "value": ["GBP","EUR","USD"] }
  ],
  "exports": ["clean_csv", "errors_csv", "summary_print"],
  "promotionSignals": [
    "Multiple reviewers need to share a batch",
    "Validated invoices must be stored centrally",
    "Audit history is required",
    "Supplier records should be reused across batches"
  ],
  "archetype": "import-validate-store"
}
</script>
```

**Required fields:** `name`, `artifactType`, `version`, `scope`,
`persistence`, `network`. Everything else is optional but strongly
recommended — their presence is what allows
[`prompts/promote-flatpack.md`](prompts/promote-flatpack.md) to
produce a useful promotion plan instead of guessing.

**`validations` vs `validation_predicates`.** The plain-text
`validations` array is the source of truth for humans. The optional
`validation_predicates` array is the structured form for receiving-side
verifiers — each predicate names a field, a constraint
(`required`, `gt`/`gte`/`lt`/`lte`, `min_length`, `max_length`,
`one_of`, `unique`, `format`, `not_in_future`, …), and, where
applicable, a value. The two forms are kept in lockstep by the author.
A Baseplate verifier can resolve predicates against actual field
declarations rather than text-matching, sidestepping the false
positives the plain-text form suffers.

**`scope`** is one of:

- `"personal"` — one user, single device. The default.
- `"shared-read-only"` — the file is **still a personal artifact**.
  It may be sent to others (over email, Slack, USB, paste-into-Notion),
  but it has **no shared state**, **no collaboration**, **no central
  authority**, and **no expectation that different recipients are
  looking at the same data**. Each recipient has their own local copy.
  Think of it as sharing a PDF or spreadsheet attachment, not as
  sharing a system.

`"shared-read-only"` is **not** a step toward Baseplate. It is the
same product as `"personal"`, distributed differently. If the answer
to "are these people expected to see synchronised data?" becomes yes,
that is a promotion event — see §8. There is no `"shared"` value;
that case is Baseplate.

**`runtime.minimumBrowser`** declares the browser floor required by
APIs the file uses. Sane defaults if omitted: Chrome 100, Safari 15.4,
Firefox 98, Edge 100 (covers `<dialog>`, `structuredClone`,
`navigator.clipboard.writeText`, ES2022). Bump it if you adopt newer
APIs.

**`archetype`** is a whole-app description, free-text, intended to help
a promotion-time agent map the Flatpack into a real backend project.
It is **deliberately not** the name of any Baseplate recipe — Baseplate
recipes are modifications layered on a base app; archetypes describe
an entire kind of app. The promotion agent translates an archetype
into the recipe set it implies. Conventional values used in this repo:

- `internal-tool`
- `import-validate-store`
- `decision-log`
- `case-workspace`
- `workflow-with-checklist`
- `branching-questionnaire`

These are suggestions, not a closed vocabulary. Coin a new one if it
fits your tool.

**Schema:** the canonical JSON Schema for the manifest is published at
[`manifest.schema.json`](manifest.schema.json). It is documentation,
not a runtime dependency — the Flatpack does not load or validate
against it. A generating LLM or a reviewing human can use it to verify
shape.

The manifest must stay consistent with `APP_META`, the `SCHEMA` block,
and the `VALIDATION` rules. If those change, the manifest changes with
them. Convention: bump `version` in all places together.

### 6.2 The HELP block

A single HTML comment near the top of the file, marked
`<!-- FLATPACK:HELP -->`, gathers two things that would otherwise be
scattered or duplicated:

```html
<!-- FLATPACK:HELP

  Flatpack:     <name>
  Version:      <semver>
  Purpose:      <one sentence>
  Data:         <local-only | none>
  Network:      <none | user-supplied-endpoint>
  Persistence:  <none | localStorage | indexedDB>

  HUMAN HELP
    Inputs:   <what the user provides>
    Outputs:  <what they get back>
    Limits:   <known constraints>

  AI EDITING NOTES
    - Safe to modify:  CORE_LOGIC, SCHEMA, VALIDATION, RENDERING, STYLES.
    - Edit with care:  STATE — migrate localStorage if the shape changes.
    - Avoid modifying: IMPORT_EXPORT unless changing file formats.
    - Rules: no network, no external dependencies, no embedded secrets.
    - Run TEST_CASES (?test=1) after changes to CORE_LOGIC or VALIDATION.
-->
```

Predictable location, single grep. Agents reading the file look here
first; agents modifying the file update it here.

Plus, in the UI:

- The tool's name in the page title and an `<h1>`.
- A one-sentence purpose immediately under the title.
- A visible privacy banner that matches `APP_META.persistence` and
  `APP_META.network`.
- A "Load sample data" affordance so the tool is usable on first open.
- A visible primary action.
- A "Reset" affordance.
- A `#print-report` section with print styles that suppress controls.
- At least one runnable `TEST_CASES` entry, executed via `?test=1`.

Plus, in the code:

- A single `APP_META` constant. Header, banner, and report read from it.
- A single `validate()` function. All input paths use it.
- A single `render()` function. All UI updates go through it.
- Core logic that is pure — no DOM access, no state mutation.

## 7. How does an AI agent modify a Flatpack safely?

The `FLATPACK:HELP` block (§6.2) is the single source of truth for
which sections are safe to modify. Read it first.

An agent editing a Flatpack must:

1. Stay inside the marked sections. If a change needs code in a
   different region, it requires an explicit instruction, not a
   reshuffle for tidiness.
2. Refuse to add dependencies, network calls, or telemetry. If the
   user asks for any of those, stop and explain that the tool is no
   longer a Flatpack.
3. Bump `APP_META.version` — patch for fixes, minor for new behaviour,
   major for any breaking shape change in schema or persistence.
4. Migrate persisted state if the shape changes. Either bump the
   `STORAGE_KEY` version suffix and start fresh, or write a
   `loadState()` migration step. Never silently corrupt a user's data.
5. Add a `TEST_CASES` entry for any new core-logic behaviour. Run
   `?test=1` and confirm pass before declaring done.
6. Re-walk [`QUALITY_CHECKLIST.md`](QUALITY_CHECKLIST.md) on anything touched.

See [`prompts/modify-flatpack.md`](prompts/modify-flatpack.md) for the
brief template.

## 8. When should it be promoted to Baseplate?

A Flatpack is **my tool**. Baseplate is **our tool**.

Promote a Flatpack into a Baseplate project when the answer to any of
these turns into "yes":

- Does a second person need to use this with shared state?
- Is anyone treating the data inside it as a source of truth?
- Does the use case need an audit log the end user cannot tamper with?
- Does it require accounts, roles, or permissions?
- Does it require a server-side API with a secret key?
- Is the cost of losing the file becoming bigger than the cost of
  rebuilding it?

**Flatpack is designed to be promoted, not automatically converted.**

A Flatpack is one HTML file; a Baseplate project is a full stack.
There is no button that transmutes one into the other. What promotes
is the *understanding* embedded in the file — and the manifest
(see §6.1) is what makes that understanding machine-readable.

### 8.1 What carries over

| Flatpack | Becomes in Baseplate |
|---|---|
| Input fields / manifest entities | Database models, API schemas |
| Local state | Persistent tables |
| `VALIDATION` rules | Backend + frontend validation |
| `CORE_LOGIC` functions | Shared service / module |
| `IMPORT_EXPORT` helpers | API endpoints, background jobs |
| Sample data | Seed fixtures, integration test inputs |
| `TEST_CASES` | Unit / integration tests |
| UI flow and rendering | Baseplate screens |
| Local-only privacy note | Production data + security model |
| `promotionSignals` in manifest | Scoping decisions in the scaffold plan |

### 8.2 The promotion flow

The full agent flow lives in
[`prompts/promote-flatpack.md`](prompts/promote-flatpack.md).
Summary:

1. **Decide.** The user notices a promotion trigger above.
2. **Analyse.** An agent reads the Flatpack — manifest first, then
   `SCHEMA`, `VALIDATION`, `CORE_LOGIC`, `IMPORT_EXPORT`, sample data,
   `TEST_CASES` — and extracts the working understanding.
3. **Plan.** The agent produces a Baseplate **promotion plan**:
   entities, roles, required features, recipe choice, open questions.
4. **Scaffold.** Baseplate (or the user) generates the new project
   from that plan. The Flatpack is *not* the codebase. It is the spec.
5. **Preserve.** The original Flatpack moves into the new project as
   reference material:
   ```
   /reference/original-flatpack.html
   /reference/sample-data.csv
   /reference/promotion-plan.md
   ```
   It documents the workflow before it became production software,
   and it stays useful for fast iteration and side-by-side
   verification.

### 8.3 The honest claim

> **Flatpack is not a smaller Baseplate. It is the proving ground
> before Baseplate.**
>
> When a tool is just for you, keep it flat. When people start
> depending on it, give it a base.

## Appendix A — Why these constraints

The constraints exist to make Flatpacks survive three predictable
threats:

- **Time.** A tool you open six months later should still work. No
  expired CDN URLs, no removed npm packages, no broken build chains.
- **Distribution.** A tool you email to a colleague should run on
  their machine without setup. No "you need Node," no "spin up Docker."
- **Future AI editing.** A tool you ask a different agent to modify
  later should not get re-architected. The section markers and the
  pure-core-logic discipline exist for that.

Anything that doesn't serve those three goals is decoration and can
be relaxed. Anything that does is non-negotiable.

## Appendix B — Anatomy reference at a glance

| Section | Contains | Side effects |
|---|---|---|
| `HELP` | Help text + AI editing notes | None |
| `STYLES` | CSS, print styles | None |
| `HTML` | Static DOM | None |
| `TEMPLATES` | HTML fragments | None |
| `MANIFEST` | `<script type="application/json">` block | None |
| `METADATA` | `APP_META` constant | None |
| `SCHEMA` | Field/record shape | None |
| `STATE` | `state`, `loadState`, `saveState` | localStorage / IndexedDB |
| `VALIDATION` | `validate(data, schema)` | None (pure) |
| `IMPORT_EXPORT` | File I/O helpers | Reads/writes user-chosen files |
| `CORE_LOGIC` | Pure functions | None |
| `RENDERING` | DOM updates from state | DOM only |
| `EVENTS` | Handlers | Mutate state, call render() |
| `TEST_CASES` | Inline tests | None |
| `BOOT` | Startup | DOMContentLoaded |
